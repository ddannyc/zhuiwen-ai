# 跨境电商运营规范知识库:元数据 Schema 与 RAG 数据管线设计

适用平台:Amazon、TikTok Shop、Temu、Shein(希音)、Ozon、Mercado Libre(美客多)

设计目标:在六个平台、多站点、多语言、规则高频变动的前提下,构建一个**可检索、可溯源、可更新、低幻觉**的运营规范知识库,既能支撑自助问答,也能为后续自动化合规校验提供结构化数据。

---

## 第一部分:统一规则元数据 Schema

核心思路:无论规则来自哪个平台、哪种来源,最终都归一化为同一条"规则记录"(rule record)。检索质量的上限由这套元数据决定,因此字段设计优先级最高。

### 1.1 字段定义

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `rule_id` | string (UUID) | 是 | 全局唯一主键 |
| `platform` | enum | 是 | `amazon` / `tiktok_shop` / `temu` / `shein` / `ozon` / `mercado_libre` |
| `site` | string | 是 | 站点/国家代码,如 `US`/`UK`/`BR`/`MX`/`RU`/`SA`。全球通用规则填 `GLOBAL` |
| `original_language` | string | 是 | 原文语言,如 `en`/`ru`/`pt`/`es`/`zh` |
| `rule_domain` | enum | 是 | 规则领域(见 1.2 受控词表) |
| `product_category` | string[] | 否 | 平台自身的商品类目,如 `Toys & Games`;通用规则留空 |
| `rule_type` | enum | 是 | `requirement`(要求)/`prohibition`(禁止)/`penalty`(处罚)/`process`(流程)/`fee`(费用) |
| `title` | string | 是 | 规则标题(改写后) |
| `summary` | string | 是 | 1–3 句结构化提炼(改写,非原文照搬) |
| `content` | text | 是 | 完整结构化内容(改写/要点化) |
| `severity` | enum | 否 | `info`/`warning`/`critical`,用于处罚类规则标注风险等级 |
| `source_type` | enum | 是 | `official_help_center`/`api_doc`/`seller_backend`/`merchant_policy`/`community` |
| `source_url` | string | 否 | 原文链接(后台/招商类可能无公开链接) |
| `version` | string | 是 | 内部版本号,如 `2026.06.01-r2` |
| `effective_date` | date | 否 | 规则生效日 |
| `expiry_date` | date | 否 | 失效日(已知则填) |
| `last_verified_at` | datetime | 是 | 最后一次人工/自动核验时间 |
| `verification_status` | enum | 是 | `verified`/`needs_review`/`outdated` |
| `confidence` | enum | 是 | `high`/`medium`/`low`,由来源可靠度决定 |
| `related_rule_ids` | string[] | 否 | 关联规则(如某要求对应的处罚条款) |
| `tags` | string[] | 否 | 自由标签,辅助检索 |
| `embedding_id` | string | 否 | 对应向量库中的向量引用 |

### 1.2 `rule_domain` 受控词表

统一的规则领域分类,跨平台共用,便于横向对比:

- `listing` 刊登规范(标题、图片、属性、A+/详情页)
- `category_qualification` 类目准入/资质审核
- `prohibited_products` 禁限售商品
- `intellectual_property` 知识产权(品牌备案、侵权)
- `logistics_fulfillment` 物流与履约(时效、面单、海外仓)
- `returns_refunds` 退换货
- `after_sales` 售后与客诉
- `account_health` 账号健康/绩效指标
- `penalties` 违规与处罚
- `fees` 费用与佣金结构
- `advertising` 广告与营销规范
- `tax_compliance` 税务与合规(VAT、EPR、清关)
- `payments` 收款与结算

### 1.3 `confidence` 与 `source_type` 的映射建议

来源直接决定可信度,生成回答时据此决定语气和提示强度:

- `official_help_center` / `api_doc` → `high`
- `seller_backend`(后台通知截图/整理)→ `medium`
- `merchant_policy`(招商政策文档)→ `medium`
- `community`(社群/经验沉淀)→ `low`,回答时必须提示"非官方,供参考"

## 第二部分:RAG 数据管线

整体分十个阶段,前五个是"数据入库",后五个是"问答服务"。

### 2.1 数据入库管线(Ingestion → Index)

**阶段 1:采集(Ingestion)**
每个平台一个独立 connector,按来源类型分流:
- API 类(Amazon SP-API、Ozon、Mercado Libre Developers):结构化拉取类目、合规属性、违规枚举。
- 公开帮助中心:受控爬取,遵守 robots.txt、限频,仅抓事实性规则。
- 后台/招商/社群:人工录入接口(运营团队走统一表单)。

> 合规红线:禁止登录态自动抓取卖家后台;原文不整段转储,只做提炼改写。

**阶段 2:归一化与翻译(Normalization)**
- 统一编码、清洗 HTML/格式噪声。
- 非中/英文原文(俄语 Ozon、葡/西语美客多)做机器翻译,但**保留原文**(原文进 `content` 备份字段,译文供检索),避免翻译歧义导致合规误判。

**阶段 3:结构化抽取(Extraction)**
用 LLM 辅助把原始文本映射到 Schema:抽取 `rule_domain`、`rule_type`、`severity`,生成改写后的 `summary`/`content`。这一步是"原文 → 规则记录"的转换器。提示词需强制:只提炼不杜撰、无法判断的字段留空、标注不确定项。

**阶段 4:人工审核队列(Human-in-the-loop)**
所有 `confidence != high` 或 LLM 标注不确定的记录,进入审核队列由运营/合规人员确认后才置为 `verified`。这是质量护城河,尤其 Temu/Shein 几乎全程依赖此环节。

**阶段 5:分块、向量化与索引(Chunk → Embed → Index)**
- 按"条款"语义分块,保留所属标题作为上下文,避免固定字数硬切。
- 双索引:向量库(语义检索)+ 关键词/BM25 索引(精确命中类目名、违规代码、费率数字)。
- 每个 chunk 继承父规则记录的全部 metadata,供检索期过滤。

### 2.2 问答服务管线(Query → Answer)

**阶段 6:查询理解**
解析用户问题,抽取 `platform`、`site`、`product_category` 等过滤条件。例:"亚马逊美国站玩具类目能不能卖含磁铁的产品" → platform=amazon, site=US, category=Toys, domain∈{prohibited_products, category_qualification}。

**阶段 7:混合检索 + 元数据过滤(关键)**
先用 metadata filter 把候选锁定到正确平台/站点/类目,再做向量+关键词混合召回、重排序。**不做元数据过滤会导致跨平台串台**(把 Temu 的规则答到亚马逊问题上),这是规则类知识库最常见的事故。

**阶段 8:生成 + 强制溯源**
- 基于召回结果作答,每条结论附 `source_url` + `version`。
- 检索不到就明确说"未找到相关规则",**禁止编造**——给错合规建议可能导致卖家被封号,代价极高。

**阶段 9:时效守卫(Freshness Guard)**
回答前检查命中规则的 `last_verified_at` 与 `verification_status`:
- 超过设定阈值(如 30/60 天)→ 回答中附"该规则可能已更新,请以平台最新公告为准"。
- `confidence=low` → 标注"非官方信息,供参考"。

**阶段 10:反馈闭环**
用户对回答打分/纠错 → 进入审核队列 → 修正规则记录或补充缺口。形成"问答暴露盲区 → 反哺数据"的循环。

### 2.3 更新与版本机制(贯穿全管线)

平台规则高频变动,必须独立设计:
- **定时核验任务**:按平台设不同频率(亚马逊政策页周级,Temu/Shein 后台通知人工日级跟进)。
- **变更检测(diff)**:重新抓取后与上一版本做内容 diff,有变化才触发重新抽取+审核,避免无效重算。
- **版本留存**:规则不做物理删除,旧版本标 `outdated` 保留,支持"某规则历史是什么"的追溯。
- **过期标记**:超期未核验自动降级为 `needs_review`。

## 第三部分:落地优先级

资源有限时的推进顺序:

1. **先跑通文档化好的平台**:用 API + 公开帮助中心把 Amazon、Ozon、Mercado Libre 的自动化管线打通,验证整套 Schema 与 RAG 架构。
2. **TikTok Shop 分区域半自动**:按美区/东南亚/欧洲分站点维护,人工+半自动结合。
3. **Temu / Shein 靠人工团队补齐**:这两个平台规则最难获取,整理得全即是产品差异化卖点,作为护城河重点投入。

## 附:数据事故防范清单

- [ ] 每条规则可溯源(链接 + 版本)
- [ ] 检索强制带 platform/site 过滤,杜绝跨平台串台
- [ ] 原文不整段转储,只存改写提炼(版权)
- [ ] 不登录态抓取后台(合规/封号风险)
- [ ] 检索不到宁可说"不知道",不编造合规建议
- [ ] 超期规则与非官方来源在回答中明确提示
